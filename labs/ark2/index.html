Ark explainer

ELI5

  • Alice sends money to the bank and in return receives a check from the bank that she can redeem anytime anywhere (not only on this bank)
  • If Alice wants to pay Bob using Ark, she can ask the bank to pay Bob, and in return she will endorse their check back to the bank.
  • The bank can only redeem Alice's check if it shows proof of payment to Bob.
  • Bob receives a check from the bank that he can redeem anytime anywhere.

TL/DR

  • There are two parties involved in Ark: users and the Ark Service Provide (ASP) aka Server.
  • Ark implements VTXOs, and all values exchanged inside Ark use VTXOs.
  • VTXOs are outpoints from hidden transactions, that can be broadcasted to mainchain to become UTXOs, or sent to the ASP in exchange for other VTXOs.
  • Spending and receiving VTXOs doesn't touch the chain, which allows for an infinite exchange of value inside Ark with a minimal chain footprint.
  • Ark is a completely trustless solution: users don't need ASP's permission to get their money back from the Ark, and the ASP doesn't need to trust users

Onboarding

The first step is for Alice to lift her coins. This is an atomic swap where she will give up a UTXO (+ fees) to receive a virtual UTXO (aka VTXO) from the ASP at the same time. The ASP makes a special transaction, called pool transaction (aka Pool Tx), where it mixes UTXOs from many users (among other things) and creates a shared UTXO as well as many offchain VTXOs for each of those users including one for Alice.

In order to be trustless:

  • Alice sending funds to ASP and the ASP creating the VTXO needs to be atomic
  • Alice needs a way to exit the Ark without ASP's permission

Pool transaction

The ASP places one (or more) input(s) to its pool transaction to supply funds to the shared UTXO output, connectors output, change output, and transaction fees.

The shared UTXO output is a UTXO locked by 2 taproot leaves:

  • A multisig n-of-n, where n is the number of participants in the transaction.
  • The other leaf, pk(ASP) && older(4 weeks), allows the ASP to spend it alone after 4 weeks.

In order to be trustless, before signing the pool transaction Alice needs a way to unilaterally get their funds back. To achieve this, the ASP builds and signs a binary tree of transactions and shares them with each participant.

Each leaf on this tree represents a VTXO:

In the example above, Alice's VTXO will be tx1:1 => tx2:1 => tx4 that she can broadcast anytime to receive their funds back. For comparison, Bob's VTXO will be tx1:1 => tx2:0 => tx5.

Notes:

  • Since tx1:1 => tx2:1 => tx4 (the VTXO) have no fees, Alice will need to add an extra transaction to pay for all the fees (see CPFP).
  • The number of transactions on this branches will be equal to log2(n) + 1, where n = number of participants

Lifting

Frank now wants to enter the Ark, so he must use a process called lifting, which is an atomic two-way peg mechanism that doesn't require trust. Lifting lets users lift their onchain utxos off the chain for a 1:1 VTXO. Users can unilaterally redeem a VTXO for an onchain utxo without asking for ASP cooperation.

  1. Frank creates a Funding transaction:
    • adds any segwit output as an input (must be segwit)
    • adds output locked by (Frank + ASP) or (ASP in 1 month)
    • sends it to ASP, not signed
  2. Frank receives a signed Redeem transaction from the ASP:
    • spends Funding transaction (1) via Frank + ASP
    • has output locked to (Frank + ASP) or (Frank in 24h)
  3. Frank signs and broadcasts Funding transaction (1)
    • Is now safe for Frank to broadcast his funding transaction because now he can leave anytime with Redeem transaction (2) spending Frank in 24h, i.e. Frank will be able to get his funds back in 24 hours.
  4. ASP (with Frank) prepares next Pool transaction
  5. Frank creates a Forfeit transaction:
    • spends from Redeem transaction (2) via Frank + ASP
    • adds connector output from Pool transaction (4) as input
    • signs and sends it to the ASP
  6. ASP broadcasts Pool transaction (4)
  7. Frank has now a VTXO
    • A VTXO is very similar to a Redeem transaction
  8. After 1 month ASP claims Frank's funds spending Funding transaction (1) via ASP in 1 month

So far, Frank polluted onchain by funding an address, and the ASP by claiming funds from that address.

Further steps from here will be footprint minimal.

Adversarial scenarios

  • Frank broadcasts Funding transaction (1) but ASP never responds:
    Frank broadcasts Redeem transaction (2) and get his funds back in 24 hours

  • Frank broadcasts Redeem transaction (2) after Pool transaction (3):
    ASP broadcasts Forfeit transaction (5) and claims Frank's funds

  • ASP doesn't wait for confirmation of Funding transaction (1) before broadcasting Pool transaction (4):
    Frank will double spend his original UTXO, making the funding transaction invalid and them spending the VTXO, thus stealing the ASP

Payment

Alice then instructs the ASP to take her VTXO and pay Oscar. She will only need to know Oscar's address. This payment is also atomic and private. The ASP does not know which VTXO belongs to which user and blindly mixes them to produce another anonymous VTXO set. Ark's magic occurs in the fact that the post-mix VTXOs can only be claimed by the intended recipients even though the ASP has no idea who owns them. Put another way, the ASP is simply coordinating blind coinjoin rounds.

Now let's imagine Alice wants to send Oscar 0.01 BTC using Ark:

Alice forfeits its VTXO to the ASP and in return Oscar receives a VTXO

In order to be trustless:

  • Alice sending funds to ASP and the ASP creating the VTXO for Oscar needs to be atomic
  • The ASP needs to have an insurance that Alice will not try to double spend her VTXO after it funds Oscar's VTXO, so it demands a signed transaction (the forfeit transaction) from Alice. This forfeit transaction spends from Alice's VTXO and it will be broadcasted only if Alice tries to double spend her VTXO

  1. Alice asks Oscar for an address (address O)
  2. Alice informs ASP she wants to pay 0.01 BTC to address O
  3. ASP (with help of Oscar) prepares new transaction (Pool Tx 2) and show it to Alice:
  4. Alice looks at the locking script for this new UTXO and sees Oscar's public key on it
  5. Alice understands that when Pool Tx 2 confirms, Oscar will have 0.01 BTC on Ark
  6. Alice proceeds by signing a PSBT that sends her VTXO to the ASP, on the condition that Pool Tx 2 is confirmed. We will call this the forfeit transaction:
  7. ASP cannot cheat, if he wants to receive from Alice he needs to send it to Oscar, which means Alice is paying Oscar atomically.

"Only if Pool Tx 2 is confirmed"

This condition is not possible to implement today using Bitcoin script, but we can implement it using a trick:

  • Pool Tx 1 will have additional outputs, called connectors
  • This connectors have no financial value (dust limit)
  • Connectors are singlesig spendable by the ASP (pk(ASP))
  • Alice will use one of this connectors as input to the new transaction
  • Alice will sign her VTXO with SIGHASH_ALL to commit to all inputs
  • The ASP will sign the connector input

This way, the transaction where Alice gives her VTXO to the ASP is only valid if Pool Tx 2 is confirmed.

Adversarial scenarios
  • Alice doesn't sign the forfeit transaction
    The ASP doesn't include Oscar's VTXO on the next pool transaction

  • ASP never broadcasts the next pool transaction
    ASP forfeit transaction becomes invalid, which means Alice still has full control of her VTXO

Exiting the Ark

When Alice wants to exit the Ark, she arranje with the ASP to have an output (pk(Alice)) on the next Pool transaction.

Alice could also unilaterally exit, i.e. without ASP's cooperation, by redeeming hers VTXO for an onchain utxo.

In order to be trustless:

  • Alice sending the VTXO to ASP and the ASP sending funds to Alice must be atomic
  • The ASP needs to have an insurance that Alice will not try to double spend her VTXO after it funds Alice's UTXO, so it demands a signed transaction (the forfeit transaction) from Alice. This forfeit transaction spends from Alice's VTXO and it will be broadcasted only if Alice tries to double spend her VTXO

Collaborative exit

  1. Alice generates new onchain address A
  2. Alice informs ASP she wants to pay 0.01 BTC onchain to address A
  3. ASP (with help of Alice) prepares new transaction (Pool Tx 3) and show it to Alice:
  4. Alice sees extra output paying 0.01 BTC to her address
  5. Alice understands that when Pool Tx 3 confirms, she will have 0.01 BTC onchain
  6. Alice proceeds by sending her VTXO to the ASP, on the condition that Pool Tx 3 is confirmed:
  7. ASP cannot cheat, if he wants to receive the VTXO from Alice he needs to send it onchain to Alice, which means Alice is exiting the Ark atomically.

Unilateral exit

Alice can leave the Ark anytime by redeeming hers VTXO for an onchain UTXO.

The cost of this broadcast depends on several factors:

  • How many transactions are on Alice's branch?
  • How much is mempool fees (sats/vbyte) at the moment?
  • Did anyone leaved already, making some of Alice's transactions already confirmed?
Adversarial scenarios
  • Alice doesn't sign the forfeit transaction
    The ASP doesn't include Alice's UTXO as output on the next pool transaction

  • ASP never broadcasts the next pool transaction
    ASP forfeit transaction becomes invalid, which means Alice still has full control of her VTXO
